# orafce_ext

`orafce_ext` extends [orafce](https://github.com/orafce/orafce) with additional Oracle compatibility features for WarehousePG / Greenplum Database. Currently it implements the `UTL_RAW` package and the `RAW` datatype.

## Requirements

- WarehousePG 6.x (Greenplum 6) or WarehousePG 7.x (Greenplum 7)
- `orafce` extension (bundled with WarehousePG)

## Installation

Enable both `orafce` and `orafce_ext` in each target database:

```sql
CREATE EXTENSION orafce;
CREATE EXTENSION orafce_ext;
```

To drop:

```sql
DROP EXTENSION orafce_ext;
DROP EXTENSION orafce CASCADE;
```

## The RAW type

`orafce_ext` defines `RAW` as a domain over `bytea` in the `public` schema:

```sql
CREATE DOMAIN public.raw AS bytea;
```

Because it lives in `public`, it is accessible without modifying `search_path`. Unlike Oracle, which limits `RAW` to 32 767 bytes, the maximum length here is **1 GB** (the `bytea` limit).

Raw literals use PostgreSQL's hex-escape bytea syntax:

```sql
SELECT '\x616263'::raw;   -- 3 bytes: 0x61 0x62 0x63 (ASCII "abc")
SELECT '\x'::raw;         -- 0 bytes (empty)
```

## UTL_RAW

All functions reside in the `utl_raw` schema.

### Function summary

| Function | Returns | Description |
|---|---|---|
| `utl_raw.cast_to_raw(c VARCHAR2)` | `RAW` | Reinterpret a string's bytes as RAW |
| `utl_raw.cast_to_varchar2(r RAW)` | `TEXT` | Reinterpret RAW bytes as text |
| `utl_raw.length(r RAW)` | `INTEGER` | Byte length of a RAW value |
| `utl_raw.substr(r RAW, pos INTEGER [, len INTEGER])` | `RAW` | Byte substring of a RAW value |
| `utl_raw.concat(r1 RAW [, r2 RAW, ...])` | `RAW` | Concatenate RAW values |
| `utl_raw.convert(r RAW, to_charset TEXT, from_charset TEXT)` | `RAW` | Re-encode RAW data between character sets |

### cast_to_raw

```
utl_raw.cast_to_raw(c IN VARCHAR2) RETURNS RAW
```

Reinterprets the byte sequence of a text string as a `RAW` value. No character-set conversion is performed — the raw bytes of the server's internal representation are returned as-is.

Returns `NULL` if `c` is `NULL`.

The parameter type is `text`. Because `orafce` registers an implicit cast from `oracle.varchar2` to `text`, both plain text and `varchar2` literals are accepted.

**Examples**

```sql
-- ASCII string: each character is one byte
SELECT utl_raw.cast_to_raw('abc');
 cast_to_raw
-------------
 \x616263
(1 row)

-- Single character
SELECT utl_raw.cast_to_raw('A');
 cast_to_raw
-------------
 \x41
(1 row)

-- Empty string returns empty RAW (not NULL)
SELECT utl_raw.cast_to_raw('');
 cast_to_raw
-------------
 \x
(1 row)

-- NULL input returns NULL
SELECT utl_raw.cast_to_raw(NULL::text);
 cast_to_raw
-------------

(1 row)

-- Multibyte UTF-8: 你 (U+4F60) encodes to 3 bytes: E4 BD A0
SELECT utl_raw.cast_to_raw('你');
 cast_to_raw
--------------
 \xe4bda0
(1 row)

-- varchar2 input (requires oracle in search_path, or explicit oracle.varchar2 cast)
SET search_path = oracle, public;
SELECT utl_raw.cast_to_raw('abc'::varchar2);
 cast_to_raw
-------------
 \x616263
(1 row)
```

> **Note** On Greenplum 6 with orafce 3.7, use `varchar2` (without schema prefix) when `oracle` is in `search_path`. On Greenplum 7 with orafce 4.x, you can use `oracle.varchar2` explicitly.

### cast_to_varchar2

```
utl_raw.cast_to_varchar2(r IN RAW) RETURNS TEXT
```

Reinterprets the byte sequence of a `RAW` value as text. No character-set conversion is performed. The result is the `text` type (compatible with `oracle.varchar2` via `orafce`'s implicit cast).

Returns `NULL` if `r` is `NULL`.

**Examples**

```sql
-- ASCII bytes → text
SELECT utl_raw.cast_to_varchar2('\x616263'::raw);
 cast_to_varchar2
------------------
 abc
(1 row)

-- Empty RAW → empty text (not NULL)
SELECT utl_raw.cast_to_varchar2('\x'::raw);
 cast_to_varchar2
------------------

(1 row)

-- Round-trip: cast_to_raw → cast_to_varchar2
SELECT utl_raw.cast_to_varchar2(utl_raw.cast_to_raw('hello'));
 cast_to_varchar2
------------------
 hello
(1 row)

-- Multibyte round-trip
SELECT utl_raw.cast_to_varchar2(utl_raw.cast_to_raw('你好'));
 cast_to_varchar2
------------------
 你好
(1 row)
```

### length

```
utl_raw.length(r IN RAW) RETURNS INTEGER
```

Returns the number of **bytes** in `r`. This counts raw bytes, not characters — multibyte characters contribute their full byte count.

Returns `NULL` if `r` is `NULL`. Returns `0` for an empty `RAW` (`\x`).

**Examples**

```sql
-- 3-byte ASCII string
SELECT utl_raw.length('\x616263'::raw);
 length
--------
      3
(1 row)

-- Empty RAW
SELECT utl_raw.length('\x'::raw);
 length
--------
      0
(1 row)

-- A null byte (0x00) is a valid byte and is counted
SELECT utl_raw.length('\x00'::raw);
 length
--------
      1
(1 row)

-- 你 (U+4F60) is 3 bytes in UTF-8
SELECT utl_raw.length(utl_raw.cast_to_raw('你'));
 length
--------
      3
(1 row)

-- 你好 is 6 bytes in UTF-8
SELECT utl_raw.length(utl_raw.cast_to_raw('你好'));
 length
--------
      6
(1 row)
```

> **Note** The original Oracle documentation examples that show `utl_raw.length('abc'::raw)` returning `6` contain an error: `'abc'::raw` is 3 ASCII bytes and its length is `3`. A result of `6` would correspond to a 2-character Chinese string stored in 3 bytes each.

### substr

```
utl_raw.substr(r IN RAW, pos IN INTEGER, len IN INTEGER DEFAULT NULL) RETURNS RAW
```

Returns a byte substring of `r` starting at byte position `pos`, with at most `len` bytes.

**Parameters**

| Parameter | Description |
|---|---|
| `r` | Source `RAW` value. |
| `pos` | Start position (1-based). **Cannot be 0.** Positive values count from the beginning; negative values count backward from the end (`-1` = last byte, `-2` = second-to-last, etc.). |
| `len` | Number of bytes to return. If omitted or `NULL`, returns all bytes from `pos` to the end. Must be ≥ 1. If `len` exceeds the remaining bytes, the result is clamped to the end of `r`. |

**NULL handling** — returns `NULL` if `r` or `pos` is `NULL`. A `NULL` `len` is treated as "to the end" (same as omitting it).

**Error conditions** — raises an error if:
- `pos = 0`
- `pos > length(r)` (positive) or `pos < -length(r)` (negative)
- `len < 1`
- `r` is empty (`\x`)

**Examples**

```sql
-- First byte
SELECT utl_raw.substr('\x616263'::raw, 1, 1);
 substr
--------
 \x61
(1 row)

-- First two bytes
SELECT utl_raw.substr('\x616263'::raw, 1, 2);
 substr
--------
 \x6162
(1 row)

-- From pos 2 to end (len omitted)
SELECT utl_raw.substr('\x616263'::raw, 2);
 substr
--------
 \x6263
(1 row)

-- len larger than remaining bytes → clamped to end
SELECT utl_raw.substr('\x616263'::raw, 2, 100);
 substr
--------
 \x6263
(1 row)

-- Negative pos: -1 = last byte
SELECT utl_raw.substr('\x616263'::raw, -1, 1);
 substr
--------
 \x63
(1 row)

-- Negative pos: -2 = second-to-last, return 2 bytes
SELECT utl_raw.substr('\x616263'::raw, -2, 2);
 substr
--------
 \x6263
(1 row)

-- Negative pos to end (len omitted)
SELECT utl_raw.substr('\x616263'::raw, -2);
 substr
--------
 \x6263
(1 row)

-- NULL r → NULL
SELECT utl_raw.substr(NULL::raw, 1, 2);
 substr
--------

(1 row)

-- NULL len → to end
SELECT utl_raw.substr('\x616263'::raw, 1, NULL::integer);
 substr
-----------
 \x616263
(1 row)

-- Error: pos = 0
SELECT utl_raw.substr('\x616263'::raw, 0, 1);
ERROR:  pos must not be zero

-- Error: pos > length
SELECT utl_raw.substr('\x616263'::raw, 4, 1);
ERROR:  pos exceeds RAW length
```

### concat

```
utl_raw.concat(r1 IN RAW [, r2 IN RAW, ...]) RETURNS RAW
```

Concatenates any number of `RAW` values into a single `RAW` value. `NULL` arguments are **silently skipped** (Oracle behavior). If all arguments are `NULL`, returns an empty `RAW` (`\x`), not `NULL`.

Empty `RAW` values (`\x`) are kept — they contribute zero bytes but are not treated as `NULL`.

**Examples**

```sql
-- Two values
SELECT utl_raw.concat('\x61'::raw, '\x62'::raw);
 concat
--------
 \x6162
(1 row)

-- Three values
SELECT utl_raw.concat('\x61'::raw, '\x62'::raw, '\x63'::raw);
 concat
----------
 \x616263
(1 row)

-- NULL in the middle is skipped
SELECT utl_raw.concat('\x61'::raw, NULL::raw, '\x63'::raw);
 concat
--------
 \x6163
(1 row)

-- Empty element (\x) is kept, NULL is skipped
SELECT utl_raw.concat('\x61'::raw, '\x'::raw, '\x63'::raw);
 concat
--------
 \x6163
(1 row)

-- All-NULL → empty RAW (not NULL)
SELECT utl_raw.concat(NULL::raw, NULL::raw, NULL::raw);
 concat
--------
 \x
(1 row)

-- Compose a word from individual bytes
SELECT utl_raw.cast_to_varchar2(
    utl_raw.concat('\x48'::raw, '\x65'::raw, '\x6c'::raw, '\x6c'::raw, '\x6f'::raw)
);
 cast_to_varchar2
------------------
 Hello
(1 row)
```

> **Difference from Oracle** Oracle's `UTL_RAW.CONCAT` accepts at most 12 arguments. `orafce_ext` accepts any number via a variadic parameter.

### convert

```
utl_raw.convert(r IN RAW, to_charset IN VARCHAR2, from_charset IN VARCHAR2) RETURNS RAW
```

Re-encodes the bytes in `r` from the character set named `from_charset` to the character set named `to_charset`. The conversion is done by the PostgreSQL/iconv conversion engine.

Returns `NULL` if any argument is `NULL`. Raises an error if either charset name is unknown.

**Charset names**

Both Oracle-style and PostgreSQL/iconv-style names are accepted, and matching is case-insensitive:

| Oracle name | PostgreSQL/iconv name |
|---|---|
| `AL32UTF8` | `UTF8` |
| `ZHS16GBK` | `GBK` |
| `JA16SJIS` | `SJIS` |
| `JA16EUC` | `EUC_JP` |
| `KO16MSWIN949` | `UHC` |
| `ZHT16MSWIN950` | `BIG5` |

**Examples**

```sql
-- Same encoding: input bytes are returned unchanged
SELECT utl_raw.convert('\x616263'::raw, 'UTF8', 'UTF8');
 convert
----------
 \x616263
(1 row)

-- Oracle-style charset name (AL32UTF8 = UTF8)
SELECT utl_raw.convert('\x616263'::raw, 'UTF8', 'AL32UTF8');
 convert
----------
 \x616263
(1 row)

-- UTF-8 → GBK: 你 (U+4F60)
--   UTF-8:  E4 BD A0  (\xe4bda0)
--   GBK:    C4 E3     (\xc4e3)
SELECT utl_raw.convert('\xe4bda0'::raw, 'GBK', 'UTF8');
 convert
---------
 \xc4e3
(1 row)

-- Oracle-style target name (ZHS16GBK = GBK)
SELECT utl_raw.convert('\xe4bda0'::raw, 'ZHS16GBK', 'UTF8');
 convert
---------
 \xc4e3
(1 row)

-- GBK → UTF-8 (round-trip)
SELECT utl_raw.convert('\xc4e3'::raw, 'UTF8', 'GBK');
 convert
----------
 \xe4bda0
(1 row)

-- UTF-8 → GBK: 你好 (U+4F60 U+597D)
--   你: E4 BD A0 → C4 E3
--   好: E5 A5 BD → BA C3
SELECT utl_raw.convert('\xe4bda0e5a5bd'::raw, 'GBK', 'UTF8');
 convert
-----------
 \xc4e3bac3
(1 row)

-- Charset names are case-insensitive
SELECT utl_raw.convert('\x616263'::raw, 'utf8', 'UTF8');
 convert
----------
 \x616263
(1 row)

-- Empty input → empty output
SELECT utl_raw.convert('\x'::raw, 'UTF8', 'GBK');
 convert
---------
 \x
(1 row)

-- NULL input → NULL
SELECT utl_raw.convert(NULL::raw, 'UTF8', 'GBK');
 convert
---------

(1 row)

-- Unknown charset → error
SELECT utl_raw.convert('\x616263'::raw, 'UTF8', 'NOSUCHENCODING');
ERROR:  conversion between NOSUCHENCODING and UTF8 is not supported
```

> **Note** The original documentation shows `select 'abc'::raw` returning `\xe697a0e6958c` — this is incorrect. `'abc'` cast to `raw` yields `\x616263` (the three ASCII bytes for a, b, c). The value `\xe697a0e6958c` is the UTF-8 encoding of the two Chinese characters 无攌, unrelated to 'abc'.

---

## Differences from Oracle

| Behaviour | Oracle | orafce_ext |
|---|---|---|
| `RAW` maximum length | 32 767 bytes | 1 GB (`bytea` limit) |
| `UTL_RAW.CONCAT` max arguments | 12 | Unlimited |
| `UTL_RAW.CONCAT` all-NULL result | `NULL` | Empty `RAW` (`\x`) |
| `search_path` required for `RAW` type | Yes (`ORACLE` schema) | No (`public` schema) |
| `PARALLEL SAFE` on functions | Yes | Not declared (GP 6.x compatibility) |

## Version compatibility

| Feature | Greenplum 6 (orafce 3.7) | Greenplum 7 (orafce 4.x) |
|---|---|---|
| `varchar2` in `cast_to_raw` | Use `varchar2` with `oracle` in `search_path` | Use `oracle.varchar2` or set `search_path` |
| `PARALLEL SAFE` | Not supported | Not declared (safe to add manually if needed) |
| `CREATE EXTENSION IF NOT EXISTS` | Not supported | Supported |
