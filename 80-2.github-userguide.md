# Git 작업 흐름 가이드: Clone → Branch → Commit → Push → Merge

## 1. 저장소 클론

```bash
git clone https://github.com/unixfree/WarehousePG-Tip.git
cd WarehousePG-Tip
```

클론하면 자동으로 기본 브랜치(보통 `main`)가 체크아웃됩니다.

## 2. 새 브랜치 생성 및 이동

```bash
# 브랜치 생성과 동시에 이동 (권장)
git checkout -b feature/my-work

# 또는 최신 Git 방식
git switch -c feature/my-work
```

`-b`(또는 `-c`)는 브랜치를 새로 만들면서 그 브랜치로 전환합니다. 현재 브랜치 확인은 `git branch` 또는 `git status`로 할 수 있습니다.

## 3. 파일 수정 후 커밋

```bash
# 파일을 수정/추가한 뒤
git add .                    # 모든 변경사항 스테이징
# 또는 특정 파일만: git add 파일명

git commit -m "기능 추가: 설명 메시지"
```

여러 번 작업하며 커밋을 반복할 수 있습니다. 커밋 전에 `git status`로 무엇이 스테이징됐는지, `git diff`로 변경 내용을 확인하는 습관이 좋습니다.

## 4. 원격 저장소로 푸시

```bash
git push -u origin feature/my-work
```

`-u`(`--set-upstream`)는 로컬 브랜치와 원격 브랜치를 연결합니다. 한 번 설정하면 이후엔 `git push`만 입력해도 됩니다.

> 푸시하려면 저장소에 대한 쓰기 권한이 필요합니다. 권한이 없는 경우 먼저 fork한 뒤 본인 저장소로 푸시해야 합니다.

## 5. 머지 (Merge)

머지는 두 가지 방식이 있습니다.

### 방식 A — GitHub Pull Request (협업 시 권장)

푸시 후 GitHub 저장소 페이지에 가면 "Compare & pull request" 버튼이 나타납니다. PR을 생성하고 리뷰를 거친 뒤 "Merge pull request" 버튼으로 머지합니다. 협업 프로젝트에서는 이 방식이 표준입니다.

### 방식 B — 로컬에서 직접 머지

```bash
git checkout main            # 머지 대상 브랜치로 이동
git pull origin main         # 최신 상태로 업데이트
git merge feature/my-work    # 작업 브랜치를 병합
git push origin main         # 병합 결과를 원격에 반영
```

## 6. 정리 (선택)

머지가 끝나면 작업 브랜치를 삭제할 수 있습니다.

```bash
git branch -d feature/my-work              # 로컬 브랜치 삭제
git push origin --delete feature/my-work   # 원격 브랜치 삭제
```

## 충돌(Conflict)이 발생하면

머지 시 같은 부분을 다르게 수정했다면 충돌이 납니다. 충돌 파일을 열어 `<<<<<<<`, `=======`, `>>>>>>>` 표시 부분을 직접 수정한 뒤:

```bash
git add 충돌해결한파일
git commit              # 머지 커밋 완성
```

---

## 전체 흐름 요약

```
clone → branch 생성 → 수정 → add → commit → push → (PR 또는 merge) → push
```
